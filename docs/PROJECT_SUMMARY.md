# 국가유산 AI 해설사 — Hybrid RAG 시스템 정리 문서

## 1. 개요

본 시스템은 **국가유산청 Open API의 공식 데이터**를 기반으로, 어떤 질문에도 **왜곡 없이** 국가유산을 해설하고 다국어로 안내하는 AI 챗봇이다.

생성형 LLM을 단독으로 쓰면 다음 문제가 발생한다.

- 존재하지 않는 사실을 그럴듯하게 지어냄 (Hallucination)
- 건립 연도·인물·유산 간 관계 혼동
- 잘못된 전제("폭설로 무너졌다")에 끌려가 오답 생성

이를 막기 위해 본 시스템은 단순 LLM 챗봇이 아니라 **Hybrid RAG(Retrieval-Augmented Generation)** 구조를 적용한다. 답변 생성 전에 **공식 데이터를 먼저 검색**하고, **검색된 근거 안에서만** 답을 생성하며, 근거에 없으면 **"확인되지 않습니다"** 라고 답한다.

---

## 2. 기술 스택

| 구분 | 채택 | 비고 |
|---|---|---|
| 백엔드 | Python · FastAPI | REST API |
| LLM | Google Gemini 2.5 Flash / 2.5 Flash-Lite | 질문 난이도에 따라 라우팅 |
| 임베딩 | Gemini `gemini-embedding-001` (768차원) | provider 추상화로 교체 가능 |
| 벡터 DB | **PostgreSQL + pgvector** | pgvector는 별도 DB가 아닌 PostgreSQL 확장 |
| 데이터 | 국가유산청 Open API (XML) | 인증키 불필요 |
| 프론트엔드 | React 18 · Vite (SPA) | 카카오톡 스타일 챗 UI |
| 언어 | 한국어 · 영어 · 중국어 · 일본어 | |

> 벡터·개인화·요청로그 테이블이 모두 **하나의 PostgreSQL**에 있어 별도 인프라가 필요 없다.

## 3. 전체 처리 흐름

```
사용자 질문 (+ 대화 이력 + 익명 user_id)
        ↓
[캐시 확인]  ── 적중 시 → 즉시 응답 (LLM 0회)
        ↓
[질의 재작성]  지시어("그 둘") 있을 때만, 저렴 모델로 독립 검색어 생성
        ↓
[질문 임베딩]  gemini-embedding-001
        ↓
[하이브리드 검색]  벡터 + 유산명/별칭 필터 + 키워드(ILIKE)
        ↓
[주제 유산 추리기]  질문 주제와 무관한 곁다리 유산 제외
        ↓
[답변 생성]  비교/상위개념=Flash, 단순=Flash-Lite + 가드레일 + 관심사 + 맥락
        ↓
[관심 가중치 누적] + [토큰·지연 로깅]
        ↓
최종 답변 (해설 + 이미지 + 근거 출처 + 메트릭)
```

## 4. 핵심 구성요소

### 4.1 데이터 적재 파이프라인

국가유산청 API의 해설 원문을 검색 가능한 형태로 가공해 벡터 DB에 적재한다.

```
국가유산 원문 → 청크 분할(문단 기준 300자) → 임베딩(768차원) → pgvector
용어 사전 ─────────────────────────────────→ 임베딩 → pgvector
검증된 지식 메모 ──────────────────────────→ 임베딩 → pgvector
```

- 유산 분류(bcodeName)를 `category`로 저장하여 개인화 가중치에 활용
- 대량 적재(`--bulk`), 이미지·분류 백필 명령 제공

### 4.2 용어 사전 레이어 (왜곡 방지의 핵심)

해설 원문에는 *홍예문, 우진각지붕, 다포 양식* 같은 전문 용어가 많다. 시스템은
원문에서 사전 등록 용어를 탐지해 **정의를 LLM 프롬프트에 주입**한다. 또한
`단어(한자)` 패턴을 자동 추출해 **신규 용어를 LLM이 정의→사전에 자동 등록**한다(고유명사는 제외).

### 4.3 Hybrid Retrieval (검색)

벡터 검색만으로는 정확한 유산 명칭을 놓칠 수 있어 세 신호를 결합한다.

| 신호 | 역할 |
|---|---|
| Dense Vector | 표현이 달라도 의미로 매칭 (코사인 유사도) |
| 이름 필터 + 별칭 | 질문에 언급된 유산을 균형 있게 확보 (예: 남대문→숭례문) |
| 키워드(ILIKE) | 유산명·용어명 정확 매칭 보강 |

검색 후 **주제 유산 추리기**로, 질문이 콕 집은 유산(또는 명확한 단일 주제)만 남기고
유사도가 낮은 곁다리 유산은 컨텍스트·이미지에서 제외한다.

### 4.4 멀티턴 대화

후속 질문의 지시어("그 둘 중 더 오래된 건?")를 이전 대화로 해석한다. 지시어가 있을 때만
**독립 검색어로 재작성(condense)** 한 뒤 검색하고, 생성 시 대화 맥락을 함께 준다.
유산명을 직접 말한 자족적 후속은 재작성을 건너뛴다.

### 4.5 개인화 (관심사 자동학습)

익명 `user_id`(브라우저 localStorage) 기준으로, 검색된 유산의 분류를 **관심 가중치로 누적**한다.
로그인·LLM 분류 없이 학습되며, 상위 관심 분야를 답변 생성에 반영하고
관심 분야 기반 **유산 추천**을 제공한다.

### 4.6 이미지 의도 게이트

"보여줘/소개/비교/이름만" 같은 **보여줄 의도**일 때만 사진을 표시하고,
"폭설로 무너졌어?" 같은 **특정 사실 질문**에는 답만 준다. 코퍼스에 없는 **상위 개념**("원각사")은
사진을 보류하고 개별 유산을 제안한다.

### 4.7 다국어 처리

질문을 한국어 공식 데이터로 검색하고, 답변을 영어·중국어·일본어로 생성한다.
고유명사는 원어를 병기하여(예: 숭례문(崇禮門, Sungnyemun)) 번역 오류를 줄인다.

## 5. 신뢰성 — 가드레일 & 검증 사실 보강

### 5.1 생성 가드레일 (프롬프트에 1회 적용 → 전체 답변에 일괄)

- **환각 방지**: 검색된 자료의 사실만 사용, 없으면 "확인되지 않습니다"
- **전제 정정**: 질문 전제가 자료와 다르면 바로잡음 (폭설 → 방화)
- **사료·고어 안전**: 옛 문체("~이니/~이라")를 현대어로 풀고, **인과로 곡해 금지**, 원문+뜻 병기
- **표현 위생**: 내부 구조 표현("검색된 자료 N번") 노출 금지

### 5.2 검증 사실 보강 (FAQ/낭설 대응)

국가유산청 원문에 없는 **검증된 외부 사실**을 별도 메모로 적재해, 챗봇이
"확인 불가"를 넘어 **출처와 함께 정정**할 수 있게 한다.

> 예: "남대문은 일제가 만든 이름?" → "아니다. 《태조실록》(1396)에 이미 '속칭 남대문' 기록이 있다." (출처 링크 제공)

## 6. 비용 최적화 & 관측성

### 6.1 비용 최적화

- **응답 캐싱**: 단일턴·비개인화 질문 캐시 → 재요청 시 LLM 0회
- **모델 라우팅**: 비교·상위개념은 Flash, 단순 질문·재작성은 저렴한 Flash-Lite
- **condense 게이팅**: 지시어 있는 후속만 재작성 호출
- **이력 다이어트**: 최근 4턴·발화당 160자로 입력 토큰 축소

### 6.2 관측성 (정량 평가)

요청별 **토큰·지연·캐시여부·사용모델**을 `request_logs` 테이블에 기록하고 `/api/metrics`로 집계한다.
정성적 가드레일과 달리 **"최적화로 토큰·지연이 얼마나 줄었는지"를 숫자로 증명**한다.

| 측정 예 | 토큰 | 지연 |
|---|---|---|
| 캐시 적중 | 0 | 즉시 |
| 단순 질문 (Flash-Lite) | ~1,400 | ~1.9초 |
| 비교 질문 (Flash) | ~1,500 | ~8초 |

## 7. 정확도 평가

정답지(gold set)와 챗봇 답변을 **유형별 하이브리드 채점**으로 비교한다.

| 채점 유형 | 적용 | 방법 | 신뢰도 |
|---|---|---|---|
| 키워드 포함 | 명확한 사실(연도·지명·인물) | 핵심어 포함 여부 | 결정적 |
| 거부 검사 | 코퍼스에 없는 질문 | "확인되지 않습니다" 패턴 | 결정적 |
| LLM 심판 | 비교·정정 등 서술형 | 정답지 vs 답변 의미 비교 | 보조 |

- 함정·멀티홉 문항(예: 두 탑 높이 비교)으로 **변별력**을 확보
- 정답지 **대량 생성 도구**(LLM 초안 → 사람 검수) 제공
- 카테고리별·전체 정확도 산출

> 정직성 원칙: 출제자가 만든 셋·LLM 심판은 편향 가능 → **신뢰할 90%는 규모·난이도·검수를 갖춘 셋**에서 나온다.

## 8. 구현 현황

**End-to-End RAG 파이프라인 구축 완료.**

```
국가유산청 API 수집 → 청크 → 임베딩 → pgvector → 하이브리드 검색
→ Gemini 답변 생성 → 카카오톡 스타일 챗 UI
```

| 항목 | 현황 |
|---|---|
| 적재 코퍼스 | 국보·보물·사적 54종, 366 청크 (유산 310 + 용어 56) + 검증 메모 |
| 백엔드 엔드포인트 | `/api/rag` `/api/heritage` `/api/me` `/api/recommend` `/api/metrics` 등 |
| 프론트엔드 | 챗 UI · 이미지 · 근거 원문 펼쳐보기 · 4개 국어 · 메트릭 배지 |
| 평가 | 정답지 24문항 + 채점 러너 + 생성 도구 (예비 측정 완료) |
| 관측성 | 요청별 토큰·지연 로깅 + 집계 API |

## 10. 기대 효과

1. 공식 데이터 기반으로 **국가유산 정보 왜곡 방지**
2. 환각 차단 + 전제 정정 + 출처 제시로 **신뢰성 확보**
3. **다국어 관광객 접근성** 향상
4. 관심사 기반 **개인 맞춤 해설·추천**
5. **정량 지표(토큰·지연·정확도)** 로 개선을 증명하는 운영 체계
```
