# 정확도 평가 (Evaluation)

> "정확도 90% 달성" 같은 **정량 주장**을 정직하게 뒷받침하기 위한 평가 방법론과 도구.
> 정성적 가드레일([RAG.md §8](RAG.md))과 짝을 이룬다.

---

## 1. 왜 단순 비교가 안 되나

챗봇 답변은 **자유 서술**이라 정답지와 **문자열 정확 일치(exact match)가 불가능**하다.
- 정답지 "조선 태조 7년(1398)" vs 답변 "1398년에 완성" → 의미 같지만 글자 다름 → exact match 실패.

그래서 **유형별 하이브리드 채점**을 한다.

| 채점 유형 | 적용 | 방법 | 신뢰도 |
|---|---|---|---|
| `keywords` | 명확한 사실(연도·지명·인물) | 핵심어가 답변에 포함되는지 | **결정적**(높음) |
| `refuse` | 코퍼스에 없는 질문 | "확인되지 않습니다" 패턴 매칭 | **결정적**(높음) |
| `judge` | 비교·정정 등 서술형 | 다른 LLM이 정답지 vs 답변을 의미로 비교 | 보조(편향 주의) |

---

## 2. 정답지 (gold set) 구조

[backend/eval/eval_set.json](../backend/eval/eval_set.json) — 각 문항은 질문 + 채점 스펙:

```json
{ "id": "fact-sungnyemun-year", "category": "사실", "question": "숭례문은 언제 완성됐어?",
  "expect": { "type": "keywords", "must_include": ["1398"] } }

{ "id": "correct-snow", "category": "정정", "question": "숭례문은 폭설로 무너진 적 있어?",
  "expect": { "type": "judge", "gold": "아니다. 폭설이 아니라 2008년 방화 화재로 훼손되었다." } }

{ "id": "refuse-gyeongbok-height", "category": "거부", "question": "경복궁 근정전의 높이는?",
  "expect": { "type": "refuse" } }
```

카테고리: **사실 · 용어 · 비교 · 정정(낭설 바로잡기) · 추론(멀티홉) · 거부(미적재)**.

---

## 3. 실행

```bash
cd backend
python eval/run_eval.py                  # 카테고리별·전체 정확도 출력
python eval/run_eval.py --save rep.json  # 틀린 문항·사유까지 리포트 저장
```

출력 예:
```
✅ [사실] fact-sungnyemun-year
❌ [추론] trap-tap-height
     ↳ (틀린 사유)
─ 카테고리별 ─
거부 5/5 (100%)  비교 2/2 (100%)  추론 3/3 (100%) ...
═ 전체 정확도: 24/24 (100.0%) ═
```

---

## 4. 정답지 대량 생성 (반자동)

손으로 다 만들기 어려우니, **LLM이 원문 근거로 Q&A 초안을 뽑고 → 사람이 검수**한다.

```bash
python eval/gen_eval.py 15 3   # 유산 15개 × 문항 3개 초안 생성
# → eval/generated_draft.json (모든 항목 "review": true)
```

> ⚠️ **생성 초안은 그대로 쓰면 안 된다.** 반드시 사람이 사실·표현을 검토·수정하고
> `review` 플래그를 제거한 뒤 `eval_set.json`에 합친다. (LLM이 원문을 잘못 요약했을 수 있음)

## 5. 정직한 벤치마크 원칙 ⭐

높은 점수 자체보다 **"그 점수를 믿을 수 있는가"** 가 중요하다.

- **체리피킹 금지** — 챗봇이 맞힐 질문만 넣으면 100%는 당연하다. **함정·난이도 높은 문항**(멀티홉 추론, 잘못된 전제, 과도한 세부)을 반드시 섞는다. (예: `trap-tap-height` = 두 탑 높이 비교, `trap-multihop-older` = 두 유산 연대 비교)
- **규모** — 16개로는 통계적 의미가 약하다. 50~100문항 이상으로 키운다(대량 생성 도구 활용).
- **출제 편향** — 만든 사람이 출제하면 무의식적으로 쉬운 쪽으로 치우친다. 가능하면 **시스템을 모르는 사람**이 출제.
- **LLM-심판 편향** — 같은 Gemini 계열이 자기 답을 채점하면 **후하게** 줄 수 있다. judge 판정은 **사람이 표본 검수**하거나, 더 강한/다른 모델로 심판한다. 반면 `keywords`·`refuse`는 결정적이라 그대로 신뢰 가능.
- **카테고리별 보고** — "전체 92%"보다 "거부 100%, 추론 80%"처럼 **약점이 보이는** 분해가 유용하다.

## 6. 현재 상태 (참고)

시드 24문항(함정 포함)에서 24/24. 단, **이는 예비 수치**다 — 문항을 만든 사람이 출제했고
서술형은 LLM 심판을 썼다. 신뢰할 "90%+" 주장은 **5번 원칙대로 키우고 검수한** 셋에서 나와야 한다.

| [eval/gen_eval.py](../backend/eval/gen_eval.py) | 정답지 초안 대량 생성 |
