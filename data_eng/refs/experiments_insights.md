## 측정 지표 (v3)

### 검색 단계 (Retrieval)
| 지표 | 의미 | 측정 방법 | 목표 |
|------|------|---------|------|
| **Recall@3** | 정답 chunk가 Top-3에 있는가 | testset의 expected_heritage_ids 매칭 | 85% |
| **MRR** | 정답 chunk 평균 순위 | 1/rank 평균 | 0.8 |
| **Context Precision** ⭐ | 검색된 chunk 중 실제 관련 비율 | RAGAS | 80% |
| **Context Recall** ⭐ | 정답 사실이 검색에 포함됐는가 | RAGAS | 85% |

### 답변 단계 (Generation)
| 지표 | 의미 | 측정 방법 | 목표 |
|------|------|---------|------|
| **Faithfulness** ⭐ | 답변이 검색 자료에 근거 (환각) | RAGAS | 90% |
| **Answer Relevancy** ⭐ | 답변이 질문에 맞음 (동문서답) | RAGAS | 85% |
| 정확도 | 사실과 일치 | must_include 키워드 매칭 | 85% |
| Citation Accuracy | 출처가 실제 근거와 일치 | source_url 검증 | 95% |
| No-answer Accuracy | "모름" 처리율 | 데이터 없는 질문 정답 비율 | 95% |

### 의도 분류
| 지표 | 의미 | 측정 방법 | 목표 |

| **의도별 정확도** | 의도 카테고리별 응답 정확도 | 의도별 정확도 분리 측정 | 85% |

## RAGAS 도입 설정 (Stage 7)

**평가 LLM 선택**: Gemini 2.5 Flash 무료 ⭐

> ⚠️ BK 결정: 답변은 GPT-4o-mini, 평가는 Gemini 무료.
> 이유: 모색 단계 → 상대 비교가 목적. 같은 평가 모델로 일관되게 측정하면 비교는 유효.
> 나중에 절대 수치 필요하면 GPT-4o-mini로 재평가.

```python
from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_precision,
    context_recall,
)
from langchain_google_genai import ChatGoogleGenerativeAI

# Gemini 2.5 Flash 평가 LLM (무료, 일 1,500회)
llm = ChatGoogleGenerativeAI(
    model="gemini-2.5-flash",
    temperature=0,
)

result = evaluate(
    dataset,
    metrics=[
        faithfulness,
        answer_relevancy,
        context_precision,
        context_recall,
    ],
    llm=llm,
)
```

**예상 비용 비교**:

| 평가 LLM | 한 문항 | 30문항 1회 | 50회 누적 | 비고 |
|---------|--------|----------|----------|------|
| **Gemini 2.5 Flash** ⭐ | **0원** | **0원** | **0원** | 일 1,500회 무료, 187문항/일 가능 |
| GPT-4o-mini | 약 5.5원 | 약 165원 | 약 8,300원 | 절대 수치 신뢰도 높음 |
| GPT-4 (RAGAS 기본) | 약 88원 | 약 2,640원 | 약 132,000원 | 사용 금지 |

> ⚠️ RAGAS 기본값은 GPT-4. 반드시 `llm=` 파라미터로 명시 필수.
> 안 하면 의도치 않은 GPT-4 비용 발생.

## 🔥 주요 발견 (Findings)

### [발견 003] 임베딩 모델 × Vector DB 거리함수 불일치

- **일시**: 2026-05-31
- **발견 경위**: chat.py에서 모든 답변이 "정보 없음" → 디버그로 distance 측정
- **원인 분석**:
  - KoSimCSE-roberta는 **cosine 유사도** 학습 모델
  - ChromaDB 기본 거리 함수는 **L2 squared distance**
  - 정규화 X 768차원 임베딩 → distance 값 **100~170 범위**
  - `NO_ANSWER_THRESHOLD = 1.0` (cosine 가정)으로 잘못 설정
  - → 모든 검색이 임계값 초과 → 100% 차단됨
- **distance 실측값**:
  ```
  "경복궁"          → best 141 (경복궁 근정전)
  "숭례문"          → best 114 (숭례문)
  "조선시대 궁궐 추천" → best 100 (성남 봉국사 — 의미 약함)
  "에펠탑"          → best 165 (무관한 석탑)
  ```
- **임시 조치**: `NO_ANSWER_THRESHOLD = 999.0` (비활성화)
- **근본 해결 후보** (다음 실험):
  - **Exp-205a**: 임베딩 정규화 (`model.encode(..., normalize_embeddings=True)`)
    → L2 distance가 0~2 범위로 정규화됨
  - **Exp-205b**: ChromaDB collection을 cosine distance로 생성
  - **Exp-310/311/312**: threshold 튜닝 (140/150/160)
- **첫 평가 결과 (Exp-260531-1300)에서의 영향**:
  - Recall@3: 50% (낮음 — 정규화 안 된 distance로 검색이 약함)
  - 정확도: 86.7% (LLM이 약한 검색 결과를 보완)
  - → "검색 약함, LLM 보완" 패턴 정량 확인
- **발표 어필**:
  > "임베딩 모델·벡터 DB 조합의 거리 척도 불일치 발견 → 정규화 보완 실험 진행.
  > 발견 → 보완 사이클의 가치를 정량 검증."

- ❌ **filter 78.7% → 66.7%** (-12.0%p) — 폭락 ⚠️
- ✅ single 71.7% → 72.5% (+0.8%p, 거의 영향 없음)

**원인 분석**:
1. filter 폭락 케이스 (T206 "고려시대 석탑", T208 "조선시대 왕릉", T217 "1호 국보", T219 "조선 후기"):
   - era/category 필터로 검색 좁혀지면 정답 chunk가 시맨틱 검색에서 밀림
   - 후보가 너무 많은 경우 (국보 수천 개) 필터가 도움 안 됨
2. compare 향상 케이스 (T103 경복궁/종묘, T122 광화문/남대문):
   - 첫 era 매칭 → 한쪽 chunk는 안정적 확보 → 비교 답변 가능

**결론**: 룰베이스 단순 매칭은 효과 제한적. **본선에서 개선**:
- LLM 기반 메타 추출 (질문 의도 정확 파악)
- 필터 hard → soft (가중치 또는 hybrid 검색)

**발표용 임팩트 인사이트** ⭐:
> "직관적으로 효과 예상한 메타 필터가 filter 카테고리에 오히려 -12%p.
> 정량 측정 없으면 잘못된 최적화에 시간 낭비할 뻔."

### [Exp-103] 프롬프트 v2 (환각 방지 강화) ⚠️ 역효과

- 일시: 2026-05-31
- 변경: 시스템 프롬프트 v1 → v2 ("자료에 없으면 절대 추측 X, 모른다고만 답해" 강화)
- 동일 유지: 100d와 동일 (k=3, KoSimCSE, gpt-4o-mini)

**결과**: 정확도 **50.8%** (-15.7%p vs 100d) / Faithfulness **0.463** (-20.0%p) / Answer Rel. 0.463 / Context Prec. 0.456 / Context Recall **0.456** (-19.2%p) / 응답 1.41s / 답변 108자

**관찰** ⚠️:
- 전 지표 폭락. 특히 **compare 31.8% → 6.8%** 박살
- 답변 길이 161자 → 108자 → "정보 없음" 거절 폭증의 흔적
- 응답 시간 ↓는 거절이 빨라서 (정상 답변 생성보다 짧음)
- Faithfulness 역설: 환각 방지 프롬프트가 Faithfulness를 떨어뜨림 → judge가 "자료에 있는데 거절" 케이스를 낮게 평가

**왜 역효과인가**:
1. gpt-4o-mini는 v1만으로도 이미 보수적 (no-answer 100%)
2. v2는 "절대 추측 X" 규칙 때문에 **자료에 부분만 있어도 거절**
3. compare 질문은 두 유산 자료가 모두 충분히 있어야 답함 → 한쪽 불충분 시 거절 → 6.8%로 박살

**결론**: **v2 폐기, v1 유지.** 강화 프롬프트가 항상 답은 아님.

**발표용 임팩트 인사이트** ⭐:
> "엄격한 환각 방지 프롬프트가 오히려 품질을 떨어뜨림. 정량 측정 없이는 역효과를 감지할 수 없음.
> 프롬프트 엔지니어링은 직관이 아닌 데이터 기반으로."

### 관찰 2: 검색 약함, 답변은 OK — LLM의 "조합" 메커니즘 ⭐

```
"경복궁 쉽게 설명해줘" 질문에:

이유:
  LLM은 chunk 순위·신뢰도 모름
  모든 chunk + 자기 사전 지식 조합해서 답 생성
```

**답변 분석 결과 일반 지식 비중 ~50%**:

- "전통 건축 양식" ← 일반 지식

**위험성**: 지금은 사전 지식이 정확해서 답이 맞아 보이지만,
