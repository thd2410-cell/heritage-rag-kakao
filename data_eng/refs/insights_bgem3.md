# bge-m3 / pgvector 스택 인사이트 (새 기준선부터)

> **왜 새 문서?**
> 스택을 KoSimCSE/ChromaDB(실험) → **bge-m3/pgvector(운영 백엔드 형태)** 로 전환.
> 옛 절대수치(Recall 35.5% 등)는 다른 스택이라 직접 비교 불가 → **여기서 새 기준선부터 누적.**
> 옛 인사이트는 [experiments_insights.md](experiments_insights.md) 참조(방법론·발견은 유효).

> 측정 환경: 도커-프리 하니스(`experiments/validate_bge_m3.py`) — bge-m3 임베딩 + cosine 계산만.
> 검색 품질은 임베딩+거리가 결정하므로 pgvector 없이도 동일하게 측정됨(결과 전이).

## [발견 005] facet LLM 재작성은 환각을 늘린다 (2026-05-31, Step3-B)

- 측정(faithfulness_check.py, 15개): **facet 답변 Faithfulness 0.757 vs 전체본문 0.983.**
- 원인: facet 근거가 얇은데 "3문장 써라" → LLM이 빈틈을 자기지식으로 메움(동십자각 0.10, 신무문 0.30).
- → **facet을 LLM으로 "재작성"하면 환각 ↑.** 0.757 = ~24% 환각 = 해설사엔 치명적.
- **시사**: 서버의 **템플릿(evidence 문장 그대로 표시)** 은 환각 0(원문 그대로) → 충실성에선 템플릿이 우위.
  - 앞서 answer_builder 키워드를 "구식"이라 했으나, **충실성 관점에선 템플릿이 안전.** facet은 "보여주기"엔 OK, "LLM 재작성"엔 위험.
- **① 강화 프롬프트 재측정 결과**: facet **0.757 → 0.933** 회복(전체본문 0.973). "3문장 강제" 빼고 "근거 적으면 짧게+추측금지" → 대부분 1.00.
  - → **문제는 facet이 아니라 프롬프트.** facet 살릴 수 있음.
  - 단 여전히 0.933<0.973, **동십자각 0.30**(근거 극도로 얇음)은 강화로도 환각 → thin-evidence 가드 필요.
- **answer 층 결론(Step 3)**: **facet + 강화 프롬프트 채택(0.933) + 근거 너무 얇으면 전체본문 fallback/표시(③).**
- ⭐ "측정이 기능" 3번째 사례 — facet 10K 굽기 전 환각 문제 포착 + 프롬프트로 해결.

## [발견 007] 엔드투엔드 파이프라인 작동 ✅ (2026-05-31)

- 처음으로 전체 체인 통합(end_to_end.py): **질문 → bge-m3 검색 → 그 유산 facet → 관심사 답변.**
- 데모 4개 전부 성공: 검색이 맞는 유산(서울 숭례문/광화문/원각사지 등) 찾고 → facet으로 관심사 답변 충실·집중.
- → **제품 파이프라인 작동 확인.** 조각별 검증(검색100%/facet충실/framing) + **통합 검증** 완료.
- 🟡 **통합에서 드러난 버그**: 답변에 "(근거1)","(근거2,3)" 인용 번호 섞임(LLM이 evidence 번호 인용). 프롬프트에 "근거 번호 언급 금지" 추가 필요. (조각별엔 안 보이고 통합하니 드러남 → 엔드투엔드의 가치)
- ✅ **answer_final.py = 최종 답변 함수** (발견 005·006·007 반영): 엄격 grounding + 근거 적으면 짧게 + 근거번호 금지 + 빈괄호 제거 + thin-evidence(<50자) 전체본문 fallback. 재확인 결과 (근거1) 사라지고 충실·집중 답변. → **답변 생성 파이프라인 완결.**
- 🔬 **충실성 3관심사 확대 측정**(faithfulness_3interest.py, 45건, 최종 프롬프트): 전체 **0.891** (인물 0.917 / 이야기 0.890 / 건축 0.867). 건축 위주 15건의 0.933보다 표본 3배·더 방어 가능. → pitch 헤드라인을 0.89(3종)로 갱신. (건축이 최저 — LLM이 건축지식 보태려는 경향)

## [Exp-bgem3-100] bge-m3 새 기준선 (2026-05-31)

- 환경: bge-m3(1024, normalize) + cosine, 도커-프리, **vector-only**
- 데이터: heritages.json 5743 → 정제(HTML·한자 제거) → content 청크 6393개(max 800)
- testset: 80문항 중 정답 유산 있는 76문항

**결과**
```
Recall@3 46.1%  (옛 35.5% → +10.6%p)
Recall@5 50.0%  (옛 38.2%)
MRR      0.379  (옛 0.309)
카테고리  single 52.2% / compare 63.6% / filter 36.0%
best cos  min 0.425 / median 0.584 / max 0.752
```

**인사이트**
1. **bge-m3 > KoSimCSE 확정** — 동일 testset에서 Recall@3 +10.6%p, MRR ↑. 스택 전환 정당화.
2. **🔑 약점이 뒤집혔다** — 옛 스택은 compare가 약점(31.8%)이었는데, bge-m3는 **compare가 강점(63.6%, 2배)** / **filter가 새 약점(36%)**.
   - 해석: bge-m3는 의미 비교를 훨씬 잘함. 반면 filter("서울 국보", "고려 석탑")는 순수 벡터가 약함 → **메타필터(era/region/category)가 필요한 영역.**
3. **vector-only 한계** — 백엔드는 여기에 name 텍스트검색(hybrid)을 더함 → entity/single 질문은 실제 더 높을 것(→ Exp-bgem3-101에서 측정).
4. **threshold 데이터** — best cos median 0.584 → no-answer 임계값 유사도 **~0.45~0.5** 후보.

## [Exp-bgem3-101] name+content 임베딩 (2026-05-31)

- 변경(1개): 임베딩 입력 `content` → `유산명: {name}\n{content}`
- 기준선 대비: Exp-bgem3-100

**결과**
```
Recall@3 56.6%  (100 대비 +10.5%p)
Recall@5 63.2%  (+13.2%p)
MRR      0.497  (+0.118)
카테고리  single 56.5% / compare 63.6% / filter 64.0%
best cos  min 0.422 / median 0.588 / max 0.760
```

**인사이트**
1. **name 임베딩이 전 지표 ↑, 채택 확정.** Recall@3 +10.5%p.
2. **🔥 예상 밖 — filter가 36%→64% (+28%p) 폭등.** 가설은 "name은 single/entity만 도울 것"이었는데, filter가 제일 크게 올랐다.
   - 추정: 한국 유산명이 지역·종류를 품음(예: "서울 흥인지문", "경주 불국사", "○○ 석탑") → name 임베딩이 filter 질문(지역·종목·시대)에 신호 제공.
   - single은 +4.3%p로 의외로 작음.
3. **약점 지형 또 바뀜**: 이제 single(56.5%)이 최약점, filter(64%)는 평준화됨.

**→ C(메타필터) 우선순위 하락**: filter 36%를 잡으려던 게 C였는데, **name 임베딩이 이미 filter를 64%로 올림.** 메타필터의 한계효용이 줄었다. "싼 실험(B) 먼저"가 C 빌드를 아껴준 사례.

## 🚨 [발견 004] Recall 측정 지표 결함 (2026-05-31)

- **경위**: Exp-102 후 single 실패 진단(diagnose_single.py) → 데이터갭 0%, 전부 "검색문제"로 분류됐는데 실패 목록을 보니 정답 키워드가 본문 단어(조선/궁궐/신라)였음.
- **원인**: `testset.json`에 **retrieval 정답 필드(expected_heritage/heritage_id) 없음.** `expected_keywords`는 **답변 본문 체크용**(정확도/Faithfulness)이지 검색 정답이 아님.
  - 그런데 Recall@3을 `expected_keywords ∈ 검색된 유산명`으로 측정 → "경복궁"을 제대로 찾아도 이름에 "조선/궁궐" 없어서 **miss 오판**. hit도 우연 매칭이라 양방향 노이즈.
- **영향**: **Exp-bgem3-100/101/102의 Recall 수치(46/56/56%)는 무의미한 과소평가.** 옛 KoSimCSE 35.5%도 동일 결함(같은 측정).
  - 단 **상대 비교 일부는 여전히 유효**: 같은 결함 지표로 잰 100 vs 101에서 name 임베딩이 오른 건 방향성은 맞음. 하지만 절대수치·카테고리 해석은 신뢰 불가.
- **해결**:
  - 빠른: single/compare는 **질문에 유산명이 들어있음** → 질문에서 엔티티 추출해 검색결과 이름과 매칭(자동). 캐시 있어 즉시 재측정 가능.
  - 정석: testset에 **`expected_heritage` 정답 필드 라벨링**(80문항). 이후 모든 retrieval 측정 유효.
- **교훈**: "측정이 기능이다." 지표부터 고치고 튜닝. 하마터면 고장난 지표 위에서 하이브리드 계속 튜닝할 뻔.

### 교정 측정 (measure_true.py, 질문 유산명 매칭, 5743 캐시)
- **single Recall@3 = 91% (21/23)** ← 옛 깨진 지표 57%에서 폭등.
- 남은 miss 2개도 **진짜 실패 아님 = 매칭 아티팩트**:
  - "석가탑이란?" → top1 `경주 불국사 삼층석탑`(=석가탑, **별칭**이라 글자매칭만 실패). 검색은 완벽.
  - "수원화성" → top1 `수원 화성`(**띄어쓰기** 차이로 글자매칭만 실패). 정답 검색됨.
  - → **실제 single 검색 ~100%.**
- **결론: bge-m3 벡터 검색은 원래 훌륭했다.** 46/56%는 전부 측정 결함이었음.
  - **시사**: 진짜 병목은 retrieval이 아니라 **(a) 측정 지표 (b) 답변 생성/정제/facet**일 가능성. 다음은 검색 튜닝보다 **지표 정비 + 답변 품질**로.
- ⚠️ 위는 5743 캐시 기준. 데이터 10351로 재임베딩 후 재확인 필요(다음 세션).

### ✅ 지표 수정 완료 (proper_eval.py, expected_heritage 기준)
- testset에 **expected_heritage 정답 라벨** 추가(add_expected_heritage.py): single 23 + compare 18 = 41문항.
- 평가기: 정규화(띄어쓰기 수원화성=수원 화성) + 별칭(석가탑=삼층석탑).
- **지표 감사(audit) + 수정**: 결함#1 "top-k 청크 vs 유산" 발견 → 유산 단위 dedup으로 수정. compare 과소평가 교정됨.
  - 남은 caveat(저위험): #2 부분문자열 오탐, #3 수동 라벨/별칭.
- **진짜 숫자 (5743 캐시, bge-m3 name+content, 유산 dedup)**:
  - **single Recall@3 = 100% (23/23)** — 검색 완벽.
  - compare 엔티티 Recall@5 **89%** / @10 92% / 전부적중@5 **83%(15/18)**.
  - **none** (best 유사도): single 평균 0.647(최소 0.556). overseas 최대 0.477 / out_of_topic 최대 0.502 → **거절 가능**. out_of_data 최대 0.645 → **single과 겹침**(임계값으로 못 거름, 데이터/LLM거절로 해결).
    → no-answer threshold 후보 **~0.52** (해외·딴소리용). out_of_data는 별도.
- → **single 졸업. compare(83% 전부적중)가 새 실측 약점** — 2번째 유산 가끔 top-5 밖(진짜 약점, top-k/쿼리분해 실험 대상).
- **filter 지표는 정제 ⑧(메타 표준화)에 의존** — 경복궁 category가 "사적"이라 "궁궐" 매칭 불가. 메타 표준화 전엔 메타품질만 잼 → 정제 ⑧ 후로 보류.
- 산출 스크립트: add_expected_heritage.py, proper_eval.py, measure_true.py, diagnose_single.py (+ testset에 expected_heritage 영구 추가)

## [Exp-bgem3-102] 하이브리드 (벡터 + 유산명 어휘매칭) (2026-05-31)

- 변경(1개): name+content 위에 **유산명 어휘매칭 rerank** 추가 (이름매칭 청크를 벡터순으로 앞에). 백엔드 text검색 모사.
- 기준선: Exp-bgem3-101 (name+content)

**결과**
```
Recall@3 56.6%  (101과 동일, 무변동)
Recall@5 60.5%  (-2.7%p)
MRR      0.541  (+0.044)
카테고리  single 56.5%(0) / compare 59.1%(-4.5) / filter 68.0%(+4.0)
```

**인사이트 (음성 결과 = 인사이트)**
1. **하이브리드가 R@3를 못 올림.** single(타깃) 무변동. 카테고리만 재분배(filter↑ compare↓ — 이름매칭이 비교의 2번째 유산을 밀어냄).
2. MRR만 소폭 ↑ → 순위는 약간 정돈됐으나 top-3 적중은 그대로.
3. **추정 원인 2가지**:
   - (a) **구현 한계**: 이름매칭을 *벡터순*으로만 정렬 → **정확매칭 우선(exact-first)이 빠짐**. 백엔드는 `name==term`을 rank0로 둠. "경복궁"이 "경복궁 근정전"보다 위로 안 감.
   - (b) **데이터 갭 가능성**: single 실패가 *검색 문제가 아니라 5743에 해당 유산이 없어서*일 수 있음. 그럼 retrieval 튜닝으론 못 고침.

**→ 다음 (택1)**
- **진단 먼저(추천)**: 실패한 single 질문의 정답 유산이 **5743에 존재하는지** 체크. (a)냐 (b)냐 가름 → retrieval 더 팔지, 데이터 더 모을지 결정.
- exact-match-first 하이브리드로 개선 재실행 (캐시 있어 **즉시**).

**threshold**: best cos median 0.588 → no-answer 유사도 ~0.45~0.5.
