## 1. 배경 — 관심사가 사는 두 곳 + 기존 결정과의 정합 ⚠️

관심사는 시스템에서 **두 층**에 걸쳐 있다. 이걸 섞으면 `backend_coordination.md` §3에서
BK가 밝힌 입장("데이터 왜곡 반대, 표현은 LLM framing")과 모순돼 보이므로 명확히 가른다.

| 층 | 무엇을 하나 | 담당 | 비고 |
|----|------------|------|------|
| **표현 (framing)** | 검색된 내용을 관심사 말투·강조점으로 재서술 | 백엔드 `llm.py` | §3에서 BK가 선호한 경로 — **유지** |
| **검색 (retrieval) 신호** | 어떤 유산을 끌어올지 우선순위 | **관심사 태그 (이 문서)** | §3이 다루지 않은 새 층 |
| **금지** | description 원문을 분류기에 맞게 왜곡 | — | **안 함** (§3 "출처 충실" 준수) |

### 정합 원칙 4개 (이 문서의 헌법)

1. **description 원문 불변** — 태깅은 `interest_*` **별도 메타데이터 필드**만 추가. 본문 왜곡 0.
   → `backend_coordination.md` §3 "데이터는 출처에 충실" 입장 그대로 유지.
2. **태그 = 검색용 신호**, **표현은 여전히 `llm.py` framing**.
   (건축파 질문 → architecture 태그 유산 우선 검색. 답변 말투는 LLM이 framing.)
3. "데이터로 표현까지 결정"하는 방향으로 넘어가지 않는다 (= 과적합 KEYWORDS의 전철 안 밟음).
4. **`interest_meta.reason`(LLM 생성 문장)은 답변 context에 절대 주입 금지** — BK 디버깅용 메타일 뿐,
   답변 재료가 아니다. 이게 새면 아래 "종류 2 → 종류 1" 누출이 발생. 박스05 협의 항목.

> 즉 태깅은 §3과 **모순이 아니라 보완**이다. §3은 *표현* 결정, 태깅은 *검색* 결정.

### 왜 LLM 태깅이 RAG의 할루시네이션 방어와 모순되지 않는가

> 의문: "RAG는 할루시네이션을 줄이려 쓰는 건데, 데이터를 LLM으로 가공하면 모순 아닌가?"
> → 정당한 직감. 답은 **"LLM이 데이터를 건드린다"에 두 종류가 있고, 우리는 안전한 쪽만 한다**는 것.

| | 종류 1 — LLM이 **답변 내용**을 만듦 | 종류 2 — LLM이 **라벨/분류**만 만듦 |
|---|---|---|
| 위험 | ★★★ 할루시네이션 (RAG가 막으려는 그것) | ★ 라우팅 라벨 오류 |
| 우리 태깅 | ❌ 안 함 | ✅ 이것 (interest 태그) |
| 틀리면 | 거짓 사실이 답변에 박힘 | "덜 적합한 문서 검색" = **검색 관련성** 저하 |
| 성격 | 사실 왜곡, 탐지 어려움 | **경계 있음·측정 가능·복구 가능**(재태깅) |

핵심:
- **답변은 여전히 원본 국가유산청 `description`에 grounding** → 할루시네이션 방어막 유지.
- Gemini가 만든 건 "이 유산은 건축파에게 보일 만하다"는 **꼬리표**지 답변 문장이 아님.
- 태그 오류는 "사실 거짓"이 아니라 "관련성 저하" → §4 측정 + 스팟체크로 검증 후 신뢰 (**LLM-as-labeler**, 맹신 아님).
- 참고: LLM-프리 대안도 에러 모드 있음 — 규칙(A1)=과적합, 외부소스(B)=entity linking 오연결. 무LLM ≠ 무에러.

## 2. 결정 — A안 (기존 데이터에서 Gemini 분류)

### 채택: A안 (vs B안)

| 경로 | 방법 | 채택 |
|------|------|------|
| **A** | 기존 `description`+`category`에서 관심사 점수 파생 (무료 Gemini) | ✅ **채택** |
| B | 외부 소스(인물·특일·e뮤지엄)로 보강 | ⏳ A 측정 후 약한 관심사만 보강 |

근거:
- A는 **새 수집 0, 즉시 시작, 전체(627→15K) 일괄 적용, 80문항으로 측정 가능**.
- B는 entity linking 어렵고 선투자 큰데 효과 미지수 → A로 약점(특히 `people`) 확인 후 외과수술.
- 팀원 방향(§3 framing, 기존 데이터 기반)과 정렬 → 통합 마찰 최소.

### 관심사 5종 (백엔드 정렬)

| key | 의미 | 신호 예 |
|-----|------|---------|
| `architecture` | 형태·구조·축조 방식 | 기둥·지붕·양식·석축·층 |
| `people` | 관련 인물·왕·재위 | 인물명·왕·장군·재위 |
| `travel` | 위치·방문·소장처 | 주소·소재지·박물관·관람 |
| `story` | 발견·설화·일화 | 전설·일화·발견 경위 |
| `quiz` | 수치·연대·지정 등 팩트 | 크기·연대·국보 호수·지정일 |

> 5종은 **백엔드 `answer_builder.KEYWORDS` 키와 동일**하게 시작. (확장은 6번 논의)

### 스키마 (heritages.json에 추가될 3필드)

```json
{
  "id": "H01-151",
  "name": "...", "description": "...", "category": "...",   // 기존 필드 불변

  "interest_scores": {                       // ① 5종 전부 점수 (핵심)
    "architecture": 0.9, "people": 0.1, "travel": 0.4,
    "story": 0.7, "quiz": 0.6
  },
  "interests": ["architecture", "story", "quiz"],   // ② 임계값(기본 0.5) 넘은 것
  "interest_meta": {                         // ③ 출처·버전 (락인 방지)
    "source": "gemini-2.5-flash",
    "schema_version": "v1",
    "tagged_at": "2026-05-31",
    "reason": "성벽 형태·축조 방식 서술 풍부, 발견 일화 포함"
  }
}
```

**왜 5종 점수를 전부 저장하나** (단순 라벨 X):
- 임계값을 바꿔도 **Gemini 재호출 없이** `interests` 재계산 → 임계값 튜닝 공짜.
- 나중에 B안이 `people: 0.1`처럼 **낮은 것만 골라** 덮어쓰기 가능.
- `interest_meta.source`로 "이 태그는 A(gemini)가 만든 것" 추적 → B가 선택 교체.

### 방법

- 입력: `name + category + description` → 출력: 5종 점수(0~1) + 한 줄 근거 (구조화 JSON).
- 모델: **Gemini 2.5 Flash (무료, `GOOGLE_API_KEY`)**.
- 멀티라벨 (건축+설화 동시 가능). 임계값 기본 0.5.
- 스크립트: `data_eng/code/tag_interests.py` (3번 락인 원칙 준수).

## 3. 락인 방지 설계 4원칙 (A→B 전환을 막지 않기 위해)

| # | 원칙 | 효과 |
|---|------|------|
| 1 | **데이터에 저장, 코드에 안 박음** — 백엔드는 `interests` 필드를 *읽기만* | 로직이 `tag_interests.py` 하나에 격리 |
| 2 | **Gemini 분류 사용** (규칙 키워드 아님) | 프롬프트만 고쳐 재실행 → 15K 일반화, 과적합 회피 |
| 3 | **출처·신뢰도 기록** (`source`·`schema_version`·점수) | B가 약한 태그만 선택 교체 가능 |
| 4 | **재실행 가능(idempotent)** — 다시 돌리면 깨끗이 재생성 | 데이터 태그 = 되돌릴 수 있음 (락인 아님) |

> 고치기 어려운 건 "코드에 박힌 로직"이지 "재생성 가능한 데이터 필드"가 아니다.
> 80문항 testset이 안전망: 태깅 → 측정 → 나빠지면 이전 버전으로 재생성(롤백).

## 6. ⭐ 관심사 다양성·구체화 논의 (OPEN — 진행 중)

> "백엔드 5종 그대로"는 **출발점**이다. 15K 전 종목의 다양성을 5종이 담는지는 미해결.
> 아래는 결정이 아니라 **논의 후보**. 채택 시 6번 항목을 결정으로 승격하고 변경 이력 기록.

### 6-1. 5종이 충분한가? (커버리지 의문)

- 5종은 사실상 **진흥왕 순수비(비석)** 맥락에서 출발한 백엔드 키워드 기반 → **편향 가능성**.
- 15K엔 자연유산(명승·천연기념물), 무형유산(춤·음악·기능), 민속·생활문화 등 **5종으로 안 잡히는 결**이 많음.

### 6-2. 세분화/추가 후보 (브레인스토밍)

| 후보 카테고리 | 왜 | 기존 5종과 |
|---------------|----|-----------|
| `nature` (자연·경관) | 명승·천연기념물은 architecture/people 안 맞음 | 신규 |
| `religion` (신앙·종교) | 사찰·종묘·서원 등 의례·사상 | story 일부 흡수 |
| `life` (생활·민속) | 의식주·세시풍속·공예 | 신규 |
| `art` (예술·미감) | 회화·도자·서예의 미적 감상 | architecture와 분리? |
| `architecture` 세분 | 목조/석조/성곽 → 너무 거칠 수 있음 | 하위 태그 |

### 6-3. 진화·버전 관리 방안

- 카테고리 추가/변경 시 **`interest_meta.schema_version` 올림** (v1→v2) → `--force` 재태깅.
- 백엔드 정렬 깨지지 않게 **추가는 가능, 기존 key 의미 변경·삭제는 백엔드 협의**.
- `interest_scores`에 5종 전부 저장하므로 **신규 key 추가는 재태깅만**, 기존 데이터 구조 안 깨짐.

### 6-4. 열린 질문 (다음 논의 안건)

- [ ] 멀티라벨 몇 개까지 허용? (상위 2개 vs 임계값 전부)
- [ ] 카테고리를 **종목(category) 매핑**으로 일부 자동 부여할까? (예: 천연기념물 → nature 자동)
- [ ] 연령대(age_group) 개인화와 관심사를 **교차**할 필요가 있나? (`llm.py`엔 둘 다 있음)
- [ ] 5종 유지 vs 확장을 **80문항 측정으로 판가름**할 수 있나? (측정 가능성 자체가 의문)

### 6-5. 1차 측정 결과 — `quiz`는 문서 태그 부적합 (실증, 2026-05-31)

종목 10종 30건 스팟체크 (Gemini 2.5 Flash). **태그 정확·일반화 ✅**:
자연유산=건축0·인물0, 무형유산=건축0, 비석=인물 중심 등 **변별 잘 됨**, 근거도 본문 충실(환각 없음).
→ 태그 신뢰성 확인, 다음 단계 진행 가능.

**핵심 발견 — `quiz`가 임계값 무관 거의 항상 켜짐**:

| 임계값 | quiz 발화율 | 평균 태그수/건 |
|--------|-------------|----------------|
| 0.5 | 96% (29/30) | 3.5 |
| 0.6 | 93% | 3.3 |
| 0.7 | 90% (27/30) | 2.9 |

- quiz="수치·연대·지정"인데 **모든 유산이 지정정보·연대 보유** → 항상 참. **threshold로 못 거름**(0.7에도 90%).
- architecture·people·travel·story는 점수가 0.0/1.0 **양극** → 이진 태그 적합, 변별 잘 됨.

**결정 후보(권장)**:
1. **`quiz`를 문서 태그에서 제거.** quiz는 유산 속성이 아니라 **"퀴즈 모드"라는 사용 방식**(모든 문서 해당).
   단 **백엔드 user-interest `quiz`는 유지**(llm.py 답변 모드). 즉
   **문서 태그 4종(architecture/people/travel/story) ≠ user-interest 5종** 으로 분리.
   → §1 정합 원칙과 일관: *태그=검색 신호 / quiz=표현 모드*.
2. threshold 0.5~0.6 무난 (quiz 제거 후 평균 ~2.5개). 점수 양극이라 값에 둔감.
3. **점수(0~1)를 이진 태그보다 랭킹으로** 쓰면 더 정밀 (백엔드 협의 옵션).

→ 6-4 일부 해소: "5종 유지?" → **문서 태그는 4종**(quiz 분리). "category 자동매핑?" → nature 등은 6-2 후보 유지.
