> 답변 품질에 영향을 주는 **데이터 스키마·정제 규칙**을 확정한 기준 문서.

>
> **결정 시점**
> 정제 코드 작성 전. 한 번 정제해두면 되돌리기 비쌈.

```

예시:
```

## 결정 사항 (8개)

각 항목 BK 의견 + **BK 결정** + 근거.

### 결정 근거 등급 (왜 임의가 아닌가)

> 8개 결정은 근거 출처가 다르다. "전부 실험으로 검증"이 아니라 **측정 가능한 건 측정,
> 제약은 제약대로, 관례는 관례임을 밝히고, 못 정한 건 데이터 보고 결정으로 보류**.
> 이 등급 구분 자체가 "데이터 기반 의사결정" 발표 자산.

| 등급 | 의미 | 해당 결정 |
|------|------|----------|
| 🔬 **실측 근거** | 우리 실험 수치로 뒷받침 | ⑧ (Exp-104 −12%p로 A 폐기), ② (Exp-100d/102 베이스라인), ⑤ (Exp-001 "부속 묻힘" 발견) |
| 🔒 **제약 근거** | 바꿀 수 없는 외부 조건 | ⑦ (카카오 1000자 제한), ③ (원본 API에 HTML 실재) |
| 📐 **관례·표준** | 측정 안 했으나 RAG 업계 표준 채택 | ① (description=임베딩/나머지=메타), ④ (한자 별도 필드) |
| ⏳ **데이터 후 결정** | 분포 보고 정하려 정직하게 보류 | ⑥ (결측치 임계값) |

> ⚠️ 아래 본문·영향표의 `±N%p` 중 실험으로 측정한 건 ⑧뿐. 나머지(②③④⑥)의 수치는
> **예상치(미측정)** 이며, 필요 시 Exp 1회로 관례→실측 승격 가능.

### ① 어떤 필드를 RAG에 포함할 것인가

**BK 의견**: 아래 필드 권장. documents/metadata 분리.

| 필드 | 위치 | 포함 |
|------|------|------|
| description | documents | ✅ 본문 (임베딩 대상) |
| name | metadata | ✅ 검색 핵심 |
| name_hanja | metadata | ✅ 다국어 P1·외국인 검색용 (별도 필드, ④ C안과 일치) |
| location | metadata | ✅ |
| era | metadata | ✅ |
| region | metadata | ✅ |
| category | metadata | ✅ |
| image_url | metadata | ✅ 카카오 BasicCard |
| narration_url | metadata | ✅ 음성 해설 (사회적 약자) |
| source_url | metadata | ✅ 출처 링크 |
| designation_date | metadata | ✅ metadata 전용 (답변 본문엔 안 씀) |
| **parent_id** | metadata | ✅ ⑤와 연결 (메인-부속 관계) |
| **fetched_at** | metadata | ✅ 수집 시점 (모니터링·갱신 추적) |

**BK 결정**: **B안 — documents 1개(description) + metadata 12개**

근거:
- documents 단일(description)로 임베딩 깔끔 → 검색 정밀도
- 부가 정보는 모두 metadata → 응답·필터·다국어 매핑 활용
- parent_id·fetched_at는 운영 메타로 ⑤·박스 04와 직결

### ② chunk 단위를 어떻게 나눌 것인가

**BK 의견**: 유산 1개 = chunk 1개 (현재 indexer.py 방식).

**옵션**:
- **A** (현 운영): 유산 1개 = chunk 1개 (Exp-100d/102 베이스)
- **B**: 긴 설명(궁궐 등) chunk_size=400 자연 분할
- **C**: 필드별 분리 (description / location+era / category)

**BK 결정**: **A 운영 + B/C 비교는 Exp-2XX 분화 카탈로그**

근거:
- 현 Exp-100d/102가 모두 A 베이스라 일관성 유지
- B/C는 재인덱싱 비용 있어 검증 후 전환 판단
- 분화 카탈로그에 두면 "체계적 비교 계획" 발표 어필 포인트

### ③ HTML 태그·특수 문자 처리

원본에 섞여 있음:
```
<b>경복궁</b>은 ...
&lt;국보 1호&gt;
&#x3000; (전각 공백)
```

**BK 결정**: **A — 전부 정제**

처리:
- HTML 태그 전부 제거 (`<b>`, `<i>`, `<br>` 등)
- HTML 엔티티 디코딩 (`&lt;` → `<`)
- 전각 공백 → 일반 공백
- 연속 공백 → 단일 공백

근거: 정제 안 하면 답변에 그대로 노출 → 환각률 ±2%p **(예상, 미측정)** 영향. 코드 ~10줄 (BeautifulSoup) 비용으로 확실한 품질 개선.

### ④ 한자 처리

`경복궁(景福宮)`, `종묘(宗廟)` 같은 한자 병기 처리.

**옵션**:
- A. 한자 완전 제거
- B. 한자 병기 유지
- C. 한자명을 별도 필드로 분리

**BK 결정**: **C — 별도 필드 분리**

처리:
- description에서 한자 병기 제거 → `경복궁은...` (자연스러움)
- `name_hanja` metadata 필드에 한자만 보존 → `景福宮`

근거: ① B와 자동 일치. description은 자연스러움 ↑, 한자는 다국어 P1(외국인 한자 검색)에서 활용. 두 마리 토끼.

### ⑤ 동일 유산 중복 처리

정황:
```
"경복궁"       → 4대궁 API 1건
"경복궁 근정전" → 국보 별도 1건
"경복궁 광화문" → 보물 별도 1건
"경복궁 흥례문" → 부속 별도 1건
```

Exp-001 발견: 부속 123개에 메인 5개가 묻혀 검색 안 됨.

**옵션**:
- A. 별도 보존 + `parent_id` 수동 매핑
- B. 메인만 인덱싱, 부속은 description에 통합
- C. 별도 보존, parent_id 없음

**BK 결정**: **A — 별도 보존 + parent_id 수동 매핑 (4대궁+종묘 5부모)**

처리:
- 모든 record 그대로 별도 보존 (627개 유지)
- 4대궁+종묘 5개 부모에 대해 자식 ~150개 수동 라벨링 (1~2h)
- metadata의 `parent_id`에 부모 record ID 기록
- `parent_id` 활용 메인 우선 추출 로직은 Exp-2XX 분화 카탈로그 후속

근거:
- 데이터 손실 0, 메타데이터 보존 100%
- 부속 직접 질문(`근정전이 뭐야?`) 처리 가능
- B는 description 폭증·메타 손실·되돌리기 어려움
- A는 되돌리기 쉬움 (parent_id 빼면 C와 동일)

### ⑥ 결측치 처리

description이 비어있거나 너무 짧은(예: 50자 미만) 유산.

**BK 결정**: **C — TBD, 새 JSON 수집 완료 후 데이터 분포 보고 결정**

이유:
- 2026-05-31 현재 JSON 수집 재구성 중
- 수집 단계에서 일부 유형 문화재가 이미 누락되는 현상 발견

  2. 임계값 후보 비교 검토
  3. 본 칸에 확정값 채움

**잠정 방침**: 50자 미만은 인덱싱 제외, name·location 등 metadata만 보존 (단 임계값은 변경 가능).

### ⑧ 메타데이터 표준화 (Metadata Filtering용)

```
era 표준화: "조선 시대", "조선시대", "朝鮮", "조선왕조" → "조선"
region 표준화: "서울특별시 종로구", "서울 종로구" → "서울" (광역시·도 단위)
category 표준화: "궁궐", "궁", "왕궁", "법궁" → "궁궐"
```

**Exp-104 결과**: 룰베이스 메타 필터 적용 → filter 카테고리 정확도 **-12%p**. 단순 매핑 사전이 오히려 검색을 좁힘.

**옵션**:
- A. 단순 매핑 사전 (룰베이스) — **Exp-104 실패, 폐기**
- B. LLM 메타 추출 (Exp-204 후속 검증 필요)
- C. 메타 표준화 안 함, description에 통합

**BK 결정**: **C 운영 + Exp-204에서 B 검증 후 전환 판단**

근거:
- 현 Exp-100d/102가 사실상 C 상태에서 70.6% 도달
- A는 Exp-104 -12%p로 폐기 확정
- B는 LLM 비용·정확도 검증 안 됨 → Exp-204 결과 보고 전환 판단
- 분화 카탈로그(우선순위 1)에 명시: "Exp-204 LLM 메타 추출"

## RAG 평가 입장에서 본 영향

> 📐 수치 대부분은 **예상치(미측정)**. 실측은 ⑧(Exp-104)·⑤(Exp-001)뿐.

| 결정 | RAG 평가 지표 영향 | 근거 |
|------|-------------------|------|
| ④ 한자 C안 (별도 필드) | Answer Relevancy ±2~5%p (description 자연스러움 ↑) | 📐 예상 |
| ⑤ A안 (parent_id) | Exp-001 발견(부속 묻힘) 부분 해결 가능 (활용 로직 후속) | 🔬 실측(문제) |
| ⑥ 결측치 임계값 | Recall@3 ±3%p (수집 완료 후 결정) | ⏳ 예상·보류 |
| ② chunk 단위 | Faithfulness ±3~7%p (B/C 비교는 분화 카탈로그) | 📐 예상 |
| ③ HTML 정제 | 환각률 ±2%p | 📐 예상 |
| ⑧ 메타 표준화 C | filter 정확도 −12%p (룰베이스 A 적용 시) | 🔬 실측(Exp-104) |

→ 정제 방식이 발표 수치에 직접 영향. **실측/예상을 구분해 제시하는 것이 신뢰도 핵심.**

## 후속 작업 (data_eng/README.md 박스 02 진행)

```
[즉시 수행]
✅ ① 필드 결정 → B (parent_id, fetched_at 포함 12개 metadata)
✅ ② chunk 결정 → A 운영
✅ ③ HTML 정제 → 코드 적용 (BeautifulSoup ~10줄)
✅ ④ 한자 분리 → name_hanja 별도 필드
✅ ⑦ 출처 표기 → "[출처: 국가유산포털]"
🔄 ⑤ parent_id 수동 매핑 사전 작성 (4대궁+종묘, 1~2h)
🔄 ⑥ 수집 완료 후 description 분포 분석 → 임계값 결정
🔄 ⑧ Exp-204 (LLM 메타 추출) 검증 후 B 전환 판단

[분화 카탈로그 (Exp-2XX 후속)]
- ② chunk 단위 B/C 비교
- ⑤ parent_id 활용 메인 우선 추출 로직
- ⑧ Exp-204 LLM 메타 추출
```
