# 국가유산 데이터 정제 — 협의 문서
## 정제 작업 시작 전 팀에서 함께 정해야 할 사항

> **이 문서의 의도**
> 팀원이 국가유산 데이터를 본격 정제·인덱싱하기 전에,
> RAG 답변 품질에 영향을 주는 **데이터 스키마·정제 규칙**을
> 팀이 함께 결정하기 위한 협의 문서.
>
> **누가 보는가**
> - 데이터 수집·정제 담당 팀원 → 결정 사항 따라 정제
> - RAG 평가 담당 (BK) → 평가 기준으로 활용
> - 발표 자료 담당 → 데이터 출처·규모 어필 시 활용
>
> **결정 시점**
> 정제 코드 작성 전. 한 번 정제해두면 되돌리기 비쌈.

## 왜 데이터 정제 방식이 중요한가

```
같은 국가유산청 API 데이터여도
정제 방식에 따라 RAG 답변 품질이 달라짐.

예시:
  [정제 안 한 원본]
    "경복궁(景福宮)은 <b>1395년</b> 조선 태조 이성계가
     한양으로 천도하면서 새로 지은 조선왕조 제일의 법궁이다."

  [정제 결과 A — 한자 제거, HTML 제거]
    "경복궁은 1395년 조선 태조 이성계가 한양으로 천도하면서
     새로 지은 조선왕조 제일의 법궁이다."
     → 검색·답변 자연스러움

  [정제 결과 B — 한자 병기 유지]
    "경복궁(景福宮)은 1395년 조선 태조 이성계가..."
     → 한자 검색에는 유리, 자연어 답변에는 노이즈

→ 정제 방식이 RAG 평가 지표에 직접 영향
```

### ① 어떤 필드를 RAG에 포함할 것인가

**BK 의견**: 아래 필드 권장. 추가/제외 여부 팀과 협의.

| 필드 | 포함 권장 | 이유 |
|------|----------|------|
| name (유산명) | ✅ | 검색 핵심 |
| description (설명) | ✅ | 답변 본문 |
| location (위치) | ✅ | "어디 있어?" 질문 |
| era (시대) | ✅ | "조선시대 ___" 필터링 |

### ② chunk 단위를 어떻게 나눌 것인가

**BK 의견**:
- **유산 1개 = chunk 1개** 권장 (현재 indexer.py 방식)
- 이유: 국가유산 설명문이 보통 200~500자라 chunk_size=400으로 충분
- 예외: 설명문이 1,000자 넘는 유산(궁궐 등)은 자연스럽게 분할됨

대안: 필드별 분리 (name 따로, description 따로) — 검색 정밀도는 ↑, 답변 일관성은 ↓

**팀원 결정**: (회의 후 채움)

### ③ HTML 태그·특수 문자 처리

원본 데이터에 종종 섞여 있음:
```
<b>경복궁</b>은 ...
&lt;국보 1호&gt;
&#x3000; (전각 공백)
```

**BK 의견**:
- HTML 태그 전부 제거 (`<b>`, `<i>`, `<br>` 등)
- HTML 엔티티 디코딩 (`&lt;` → `<`)
- 전각 공백 → 일반 공백
- 연속 공백 → 단일 공백

### ④ 한자 처리

`경복궁(景福宮)`, `종묘(宗廟)` 같은 한자 병기.

**옵션**:
- A. 한자 완전 제거 → "경복궁은..." (자연스러움 ↑, 한자 검색 X)
- B. 한자 병기 유지 → "경복궁(景福宮)은..." (한자 검색 ✓, 답변 다소 어색)
- C. **한자명을 별도 필드로 분리** → `name`은 한글만, `name_hanja`만 한자 (둘 다 검색 가능, 답변 자연스러움)

**BK 의견**: **C안 추천**.

### ⑦ 출처(citation) 표기 방식

답변 끝에 어떻게 출처를 명시할 것인가.

**옵션**:
- A. `[출처: 국가유산포털]` (단순)
- B. `[출처: 국가유산청 (heritage.go.kr/...)]` (URL 포함, 길어짐)

**BK 의견**: **A안 추천** (간결). 실제 URL은 webLink 버튼에서 처리.

### ⑧ 메타데이터 표준화 (Metadata Filtering용)

ChromaDB 필터 검색에 쓰일 메타 값.

```
era 표준화:
  원본: "조선 시대", "조선시대", "朝鮮", "조선왕조"
  → 표준: "조선"

region 표준화:
  원본: "서울특별시 종로구", "서울 종로구", "서울시 종로"
  → 표준: "서울" (광역시·도 단위)

category 표준화:
  원본: "궁궐", "궁", "왕궁", "법궁"
  → 표준: "궁궐"
```

**BK 의견**: 매핑 사전을 만들어 표준화 필수. 안 하면 Metadata Filtering 동작 안 함.

## RAG 평가 입장에서 본 영향

| 결정 | RAG 평가 지표 영향 |
|------|-------------------|
| 한자 처리 (A/B/C) | Answer Relevancy ±2~5%p |
| 메타 표준화 | Metadata Filtering 적용 시 정확도 ±10%p |
| 결측치 처리 | Recall@3 ±3%p |
| chunk 단위 | Faithfulness ±3~7%p |
| HTML 정제 | 환각률 ±2%p |

→ 정제 방식이 발표 수치에 직접 영향.
