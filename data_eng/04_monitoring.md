# 박스 04 — 모니터링·갱신

## 목적
배포 후 데이터 상태와 외부 API 응답 변화를 추적하고 정기 갱신한다.

## 현재 상태
⬜ 미착수 — 본선 전까지는 "1회 갱신"으로 한정 (시간 부족)

## 다뤄야 할 것

| 항목 | 내용 | 본선 전 / 후 |
|---|---|---|
| 외부 API 응답 스키마 변경 감지 | 필드 사라지면 정제 깨짐 | 본선 후 |
| 신규 유산 추가 감지 | 국가유산청 신규 지정 | 본선 후 |
| 데이터 갱신 주기 | 월 1회 / 분기 1회 / 1회 | **본선 전 결정** |
| 데이터 품질 회귀 | 정제 후 quality 검증 자동화 | 본선 후 |
| 로그 모니터링 | 답변 실패율, no-answer 비율 | 본선 후 |

## BK 결정 (잠정)

```
✅ 본선 전: 1회 갱신만 (시도 종목 + 보물 resume)
⬜ 본선 발표: "운영 단계 모니터링 계획" 1슬라이드로 명시
⬜ 자동화: 본선 후 과제로 분리
```

## 본선 발표용 운영 시나리오 (작성 예정)

```
주기적 갱신 파이프라인 (가상안)
  매월 1일 → fetcher 자동 실행
  → 신규/변경 유산 감지 → 정제 → 재인덱싱
  → 평가 → 회귀 발생 시 알림

데이터 품질 보장
  - 외부 API 응답 schema 변경 감지
  - 정제 결과 통계 비교 (필드 결측률 등)
  - 평가 testset 자동 회귀 검사
```

→ **실제 자동화는 본선 후**, 본선에서는 "이렇게 운영할 계획"만 발표.

## 다음 액션

```
⬜ 운영 시나리오 1슬라이드 작성 (본선 발표 자료용)
⬜ 데이터 갱신 주기 확정 (월 1회 권장)
```
